<span class="s1">&lt;p&gt;</span><span class="s0">First of all, I will assume you have read the </span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;reference internal&quot; </span><span class="s2">href</span><span class="s3">=&quot;ChimpLineByLine.html&quot;</span><span class="s1">&gt;&lt;span </span><span class="s2">class</span><span class="s3">=&quot;doc&quot;</span><span class="s1">&gt;</span><span class="s0">Line By Line Chimp</span><span class="s1">&lt;/span&gt;&lt;/a&gt;</span>

<span class="s0">you should also understand how TomPong works, and how to make your own version.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Now, for a brief recap of the basics of pygame. A common method of organising the code for a game is to divide it into the following</span>
<span class="s0">six sections:</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;blockquote&gt;</span>
<span class="s1">&lt;div&gt;&lt;ul </span><span class="s2">class</span><span class="s3">=&quot;simple&quot;</span><span class="s1">&gt;</span>
<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">Load modules</span><span class="s1">&lt;/strong&gt; </span><span class="s0">which are required in the game. Standard stuff, except that you should</span>
<span class="s0">remember to import the pygame local names as well as the pygame module itself</span><span class="s1">&lt;/p&gt;&lt;/li&gt;</span>
<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">Resource handling classes</span><span class="s1">&lt;/strong&gt;</span><span class="s0">; define some classes to handle your most basic resources,</span>
<span class="s0">which will be loading images and sounds, as well as connecting and disconnecting to and from networks, loading save game</span>
<span class="s0">files, and any other resources you might have.</span><span class="s1">&lt;/p&gt;&lt;/li&gt;</span>
<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">Game object classes</span><span class="s1">&lt;/strong&gt;</span><span class="s0">; define the classes for your game object. In the pong example,</span>
<span class="s0">these will be one for the player's bat (which you can initialise multiple times, one for each player in the game), and one</span>
<span class="s0">for the ball (which can again have multiple instances). If you're going to have a nice in-game menu, it's also a good idea to make a</span>
<span class="s0">menu class.</span><span class="s1">&lt;/p&gt;&lt;/li&gt;</span>
<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">Any other game functions</span><span class="s1">&lt;/strong&gt;</span><span class="s0">; define other necessary functions, such as scoreboards, menu</span>
<span class="s0">handling, etc. Any code that you could put into the main game logic, but that would make understanding said logic harder, should</span>
<span class="s0">be put into its own function. So as plotting a scoreboard isn't game logic, it should be moved into a function.</span><span class="s1">&lt;/p&gt;&lt;/li&gt;</span>
<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">Initialise the game</span><span class="s1">&lt;/strong&gt;</span><span class="s0">, including the pygame objects themselves, the background, the game</span>

<span class="s1">&lt;li&gt;&lt;p&gt;&lt;strong&gt;</span><span class="s0">The main loop</span><span class="s1">&lt;/strong&gt;</span><span class="s0">, into which you put any input handling (i.e. watching for users hitting</span>

Putting the game object classes into a file called </span><span class="s1">&lt;code </span><span class="s2">class</span><span class="s3">=&quot;docutils literal notranslate&quot;</span><span class="s1">&gt;&lt;span </span><span class="s2">class</span><span class="s3">=&quot;pre&quot;</span><span class="s1">&gt;</span><span class="s0">objects.py</span><span class="s1">&lt;/span&gt;&lt;/code&gt;</span><span class="s0">, for</span>

A note on coding styles</span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;headerlink&quot; </span><span class="s2">href</span><span class="s3">=&quot;#a-note-on-coding-styles&quot; </span><span class="s2">title</span><span class="s3">=&quot;Permalink to this heading&quot;</span><span class="s1">&gt;</span><span class="s0"></span><span class="s1">&lt;/a&gt;&lt;/h3&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">The first thing to remember when approaching any programming project is to decide on a coding style, and stay consistent. Python</span>
<span class="s0">solves a lot of the problems because of its strict interpretation of whitespace and indentation, but you can still choose the size</span>
<span class="s0">of your indentations, whether you put each module import on a new line, how you comment code, etc. You'll see how I do all of this</span>
<span class="s0">in the code examples; you needn't use my style, but whatever style you adopt, use it all the way through the program code. Also try</span>
<span class="s0">to document all of your classes, and comment on any bits of code that seem obscure, though don't start commenting the obvious. I've</span>
<span class="s0">seen plenty of people do the following:</span><span class="s1">&lt;/p&gt;</span>

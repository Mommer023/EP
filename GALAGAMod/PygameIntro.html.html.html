<span class="s1">&lt;section </span><span class="s2">id</span><span class="s3">=&quot;python-and-gaming&quot;</span><span class="s1">&gt;</span>
<span class="s1">&lt;h3&gt;</span><span class="s0">PYTHON AND GAMING</span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;headerlink&quot; </span><span class="s2">href</span><span class="s3">=&quot;#python-and-gaming&quot; </span><span class="s2">title</span><span class="s3">=&quot;Permalink to this heading&quot;</span><span class="s1">&gt;</span><span class="s0">¶</span><span class="s1">&lt;/a&gt;&lt;/h3&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s4">&amp;quot;</span><span class="s0">Is Python suitable for gaming?</span><span class="s4">&amp;quot; </span><span class="s0">The answer is, </span><span class="s4">&amp;quot;</span><span class="s0">It depends on the</span>
<span class="s0">game.</span><span class="s4">&amp;quot;</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Python is actually quite capable at running games. It will likely even</span>
<span class="s0">surprise you how much is possible in under 30 milliseconds. Still, it</span>
<span class="s0">is not hard to reach the ceiling once your game begins to get more</span>
<span class="s0">complex. Any game running in realtime will be making full use of the</span>
<span class="s0">computer.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;img </span><span class="s2">alt</span><span class="s3">=&quot;../_images/intro_blade.jpg&quot; </span><span class="s2">class</span><span class="s3">=&quot;inlined-right&quot; </span><span class="s2">src</span><span class="s3">=&quot;../_images/intro_blade.jpg&quot; </span><span class="s1">/&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Over the past several years there has been an interesting trend in game development,</span>
<span class="s0">the move towards higher level languages. Usually a game is split into</span>
<span class="s0">two major parts. The game engine, which must be as fast as possible,</span>
<span class="s0">and the game logic, which makes the engine actually do something. It</span>
<span class="s0">wasn't long ago when the engine of a game was written in assembly, with</span>
<span class="s0">portions written in C. Nowadays, C has moved to the game engine, while</span>
<span class="s0">often the game itself is written in higher level scripting languages.</span>
<span class="s0">Games like Quake3 and Unreal run these scripts as portable bytecode.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">In early 2001, developer Rebel Act Studios finished their game,</span>
<span class="s0">Severance: Blade of Darkness. Using their own custom 3D engine, the</span>
<span class="s0">rest of the game is written with Python. The game is a bloody action</span>
<span class="s0">3rd person perspective fighter. You control medieval warriors into</span>
<span class="s0">intricate decapitating combination attacks while exploring dungeons and</span>
<span class="s0">castles. You can download third party add-ons for this game, and find</span>
<span class="s0">they are nothing more than Python source files.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">More recently, Python has been used in a variety of games like Freedom</span>
<span class="s0">Force, and Humungous' Backyard Sports Series.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;img </span><span class="s2">alt</span><span class="s3">=&quot;../_images/intro_freedom.jpg&quot; </span><span class="s2">class</span><span class="s3">=&quot;inlined-right&quot; </span><span class="s2">src</span><span class="s3">=&quot;../_images/intro_freedom.jpg&quot; </span><span class="s1">/&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Pygame and SDL serve as an excellent C engine for 2D games.</span>
<span class="s0">Games will still find the largest part of their runtime is spent</span>
<span class="s0">inside SDL handling the graphics.</span>
<span class="s0">SDL can take advantage of graphics hardware acceleration.</span>
<span class="s0">Enabling this can change a game from running around 40 frames per</span>
<span class="s0">second to over 200 frames per second. When you see your Python game</span>
<span class="s0">running at 200 frames per second, you realize that Python and games can</span>
<span class="s0">work together.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">It is impressive how well both Python and SDL work on multiple</span>
<span class="s0">platforms. For example, in May of 2001 I released my own full pygame</span>
<span class="s0">project, SolarWolf, an arcade style action game. One thing that has</span>
<span class="s0">surprised me is that one year later there has been no need for any</span>
<span class="s0">patches, bug fixes, or updates. The game was developed entirely on</span>
<span class="s0">windows, but runs on Linux, Mac OSX, and many Unixes without any extra</span>
<span class="s0">work on my end.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Still, there are very clear limitations. The best way to manage</span>
<span class="s0">hardware accelerated graphics is not always the way to get fastest</span>
<span class="s0">results from software rendering. Hardware support is not available on</span>
<span class="s0">all platforms. When a game gets more complex, it often must commit to</span>
<span class="s0">one or the other. SDL has some other design limitations, things like</span>
<span class="s0">full screen scrolling graphics can quickly bring your game down to</span>
<span class="s0">unplayable speeds. While SDL is not suitable for all types of games,</span>
<span class="s0">remember companies like Loki have used SDL to run a wide variety of</span>
<span class="s0">retail quality titles.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">Pygame is fairly low-level when it comes to writing games. You'll</span>
<span class="s0">quickly find yourself needing to wrap common functions into your own</span>
<span class="s0">game environment. The great thing about this is there is nothing inside</span>
<span class="s0">pygame to get in your way. Your program is in full control of</span>
<span class="s0">everything. The side effect of that is you will find yourself borrowing</span>
<span class="s0">a lot of code to get a more advanced framework put together. You'll</span>
<span class="s0">need a better understanding of what you are doing.</span><span class="s1">&lt;/p&gt;</span>

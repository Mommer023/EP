<span class="s1">&lt;p&gt;</span><span class="s0">Pygame has been around since the early 2000s, and a lot has changed since then --</span>
<span class="s0">both within the framework itself and within the broader computing landscape as a</span>
<span class="s0">whole. Make sure to check the dates on materials you read (including this guide!),</span>
<span class="s0">and take older advice with a grain of salt. Here are some common things that</span>
<span class="s0">stick out to me:</span><span class="s1">&lt;/p&gt;</span>

<span class="s0">rects) instead of </span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;tooltip reference internal&quot; </span><span class="s2">href</span><span class="s3">=&quot;../ref/display.html#pygame.display.flip&quot; </span><span class="s2">title</span><span class="s3">=&quot;&quot;</span><span class="s1">&gt;&lt;code </span><span class="s2">class</span><span class="s3">=&quot;xref py py-func docutils literal notranslate&quot;</span><span class="s1">&gt;&lt;span </span><span class="s2">class</span><span class="s3">=&quot;pre&quot;</span><span class="s1">&gt;</span><span class="s0">pygame.display.flip()</span><span class="s1">&lt;/span&gt;&lt;/code&gt;&lt;span </span><span class="s2">class</span><span class="s3">=&quot;tooltip-content&quot;</span><span class="s1">&gt;</span><span class="s0">Update the full display Surface to the screen</span><span class="s1">&lt;/span&gt;&lt;/a&gt;</span><span class="s0">, not having scrolling backgrounds,</span>
<span class="s0">or even not filling the screen with a background color every frame because pygame</span>
<span class="s0">supposedly can't handle it.  Some of pygame's API is designed to support this</span>

<span class="s1">&lt;h3&gt;</span><span class="s0">Colorkey vs. Alpha.</span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;headerlink&quot; </span><span class="s2">href</span><span class="s3">=&quot;#colorkey-vs-alpha&quot; </span><span class="s2">title</span><span class="s3">=&quot;Permalink to this heading&quot;</span><span class="s1">&gt;</span><span class="s0"></span><span class="s1">&lt;/a&gt;&lt;/h3&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">There's a lot of confusion around these two techniques, and much of it comes</span>
<span class="s0">from the terminology used.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">'Colorkey blitting' involves telling pygame that all pixels of a certain color</span>
<span class="s0">in a certain image are transparent instead of whatever color they happen to be.</span>
<span class="s0">These transparent pixels are not blitted when the rest of the image is blitted,</span>
<span class="s0">and so don't obscure the background.  This is how we make sprites that aren't</span>
<span class="s0">rectangular in shape.  Simply call </span><span class="s1">&lt;a </span><span class="s2">class</span><span class="s3">=&quot;reference internal&quot; </span><span class="s2">href</span><span class="s3">=&quot;../ref/surface.html#pygame.Surface.set_colorkey&quot; </span><span class="s2">title</span><span class="s3">=&quot;pygame.Surface.set_colorkey&quot;</span><span class="s1">&gt;&lt;code </span><span class="s2">class</span><span class="s3">=&quot;xref py py-meth docutils literal notranslate&quot;</span><span class="s1">&gt;&lt;span </span><span class="s2">class</span><span class="s3">=&quot;pre&quot;</span><span class="s1">&gt;</span><span class="s0">Surface.set_colorkey()</span><span class="s1">&lt;/span&gt;&lt;/code&gt;&lt;/a&gt;</span><span class="s0">, and pass in</span>
<span class="s0">an RGB tuple -- say (0,0,0). This would make every pixel in the source image</span>
<span class="s0">transparent instead of black.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">'Alpha' is different, and it comes in two flavors. 'Image alpha' applies to the</span>
<span class="s0">whole image, and is probably what you want.  Properly known as 'translucency',</span>
<span class="s0">alpha causes each pixel in the source image to be only </span><span class="s1">&lt;em&gt;</span><span class="s0">partially</span><span class="s1">&lt;/em&gt; </span><span class="s0">opaque.</span>
<span class="s0">For example, if you set a surface's alpha to 192 and then blitted it onto a</span>
<span class="s0">background, 3/4 of each pixel's color would come from the source image, and 1/4</span>
<span class="s0">from the background.  Alpha is measured from 255 to 0, where 0 is completely</span>
<span class="s0">transparent, and 255 is completely opaque.  Note that colorkey and alpha</span>
<span class="s0">blitting can be combined -- this produces an image that is fully transparent in</span>
<span class="s0">some spots, and semi-transparent in others.</span><span class="s1">&lt;/p&gt;</span>
<span class="s1">&lt;p&gt;</span><span class="s0">'Per-pixel alpha' is the other flavor of alpha, and it's more complicated.</span>
<span class="s0">Basically, each pixel in the source image has its own alpha value, from 0 to</span>
<span class="s0">255.  Each pixel, therefore, can have a different opacity when blitted onto a</span>
<span class="s0">background.  This type of alpha can't be mixed with colorkey blitting,</span>
<span class="s0">and it overrides per-image alpha.  Per-pixel alpha is rarely used in</span>
<span class="s0">games, and to use it you have to save your source image in a graphic</span>
<span class="s0">editor with a special </span><span class="s1">&lt;em&gt;</span><span class="s0">alpha channel</span><span class="s1">&lt;/em&gt;</span><span class="s0">.  It's complicated -- don't use it</span>
<span class="s0">yet.</span><span class="s1">&lt;/p&gt;</span>
